# Document Similarity — in-browser (WASM) prototype

This folder is a **proof-of-concept** that runs the Document Similarity tool
entirely in the web browser using [JupyterLite](https://jupyterlite.readthedocs.io/)
+ [Pyodide](https://pyodide.org/) (CPython compiled to WebAssembly). It needs
**no server, no BinderHub, and no login** — it can be hosted as a static
[GitHub Pages](https://pages.github.com/) site. Uploaded documents never leave
the user's machine.

## Why a separate build?

The production notebook depends on three packages that **cannot** run in
Pyodide:

| Package | Problem | What we did instead |
|---|---|---|
| `atap-corpus` / `atap-corpus-loader` | Pull in **spaCy** + **pyarrow** (compiled C/C++), unavailable in Pyodide | Replaced the corpus-loader UI with a small `ipywidgets` file uploader (`FileUploaderWidget`) that reads .txt/.csv/.xlsx/.zip into a DataFrame |
| `gensim` | Compiled Cython extensions, unavailable in Pyodide | Used only for `gensim.utils.tokenize` — replaced with a regex tokenizer that reproduces gensim's default behaviour exactly |
| `swifter` | Parallel `apply` via Dask/Ray — no threads in WASM anyway | Replaced `df.swifter.apply(...)` with plain `df.apply(...)` |

The MinHash / Jaccard / visualisation logic in `content/document_similarity.py`
is otherwise **unchanged** from the production module. `panel` was also dropped
(it was used for a single warning message) to keep the in-browser package set small.

The original files in the repository root are untouched — this is an additive,
parallel build.

## Layout

```
lite/
├── content/                     # files served inside JupyterLite
│   ├── document_similarity.ipynb   # the WASM notebook (generated by make_notebook.py)
│   ├── document_similarity.py      # WASM-adapted module (see table above)
│   ├── diffviz.py                  # unchanged, pure-Python
│   └── sample_documents/           # demo corpus (2 near-dup pairs + 1 unique)
├── requirements.txt             # BUILD-time tooling only (jupyterlite-*)
├── jupyter_lite_config.json     # build config (contents dir + output dir)
├── build.sh                     # one-command local build + preview hint
└── make_notebook.py             # regenerates content/document_similarity.ipynb
```

Runtime Python packages (`datasketch`, `nltk`, `bokeh`, `seaborn`, `tqdm`,
`openpyxl`) are **not** installed at build time — the first notebook cell
installs them inside the browser with `piplite`. Only `datasketch` comes from
PyPI; the rest ship with Pyodide.

## Build & preview locally

```bash
bash lite/build.sh
python3 -m http.server -d lite/_output 8000
# open http://localhost:8000/lab/index.html?path=document_similarity.ipynb
```

`build.sh` creates an isolated `lite/.venv`, installs the tooling, and runs
`jupyter lite build`. The static site lands in `lite/_output/` (gitignored).

## Deploy to GitHub Pages

A workflow is provided at
[`.github/workflows/deploy-jupyterlite.yml`](../.github/workflows/deploy-jupyterlite.yml).
In the repo settings set **Pages → Source: GitHub Actions**, then push to the
`wasm-jupyterlite-prototype` branch (or change the trigger branch in the
workflow). The site is built and published automatically. Direct link to the
tool will be:

```
https://<org>.github.io/<repo>/lab/index.html?path=document_similarity.ipynb
```

## Known limitations of the WASM build

- **Performance**: Pyodide is single-threaded and ~2–4× slower than native.
  Comfortable for hundreds–low-thousands of short documents; a 10 MB+ corpus
  will be sluggish.
- **First load**: downloads the Pyodide runtime + packages (~30–80 MB), cached
  afterwards. Requires an internet connection on first use.
- **Tokeniser**: the regex tokeniser matches gensim's *default* `tokenize`
  (alphabetic runs, digits excluded). It does not replicate gensim's optional
  `deacc`/`lowercase` flags (which the tool didn't use).
- This is a **prototype** for evaluation, not a hardened release.

## Troubleshooting

**A code change / rebuild doesn't show up; the notebook runs the old cell
(e.g. `ModuleNotFoundError` for a package the new install cell adds).**
JupyterLite saves every notebook you open into the browser's local storage
(IndexedDB) and loads *that* copy on later visits, ignoring the rebuilt file.
A hard refresh does **not** clear it. To get the fresh notebook:

1. In the JupyterLite file browser, right-click the notebook → **Delete**, then
   reload and reopen it (it falls back to the freshly-built server copy); or
2. DevTools → **Application → Storage → Clear site data**, then reload.

The bundled `.py` modules are re-fetched on each kernel start, so only the
notebook itself is "sticky". Always run the cells top to bottom — the first
cell must finish installing packages before the import cell runs.
